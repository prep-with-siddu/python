## 3. Understanding REPL (Read-Eval-Print Loop)

REPL is Python's **interactive shell** â€” type code, see output instantly.

```bash
$ python3
>>> 2 + 3
5
>>> print("Hello World")
Hello World
>>> name = "Sid"
>>> f"Hi {name}"
'Hi Sid'
>>> exit()
```

**How REPL works:**
1. **Read** â€” Takes your input
2. **Eval** â€” Evaluates/executes the expression
3. **Print** â€” Displays the result
4. **Loop** â€” Goes back to step 1

**When to use REPL:**
- Quick testing / experimenting
- Checking how a function works
- Debugging small snippets

> ğŸ’¡ Type `help(function_name)` in REPL to see documentation. Example: `help(print)`

## 5. Understanding `pip` and `pip install`

**`pip`** = Python's package manager (like npm for Node.js).


## 6. Virtual Environments (`venv`, `virtualenv`)

### Why Virtual Environments?
- Each project may need **different versions** of the same package.
- Without venv: all packages install **globally** â†’ version conflicts!
- With venv: each project has its **own isolated** set of packages.

### Using `venv` (Built-in, Python 3.3+)
```bash
# Create a virtual environment
python3 -m venv myenv

# Activate it
source myenv/bin/activate      # macOS/Linux
# myenv\Scripts\activate       # Windows

# Your terminal now shows: (myenv) $
# Everything you pip install goes into myenv/ only

# Install packages inside venv
pip install requests
pip install flask

# Deactivate when done
deactivate
```

### Directory Structure After Creating venv
```

â””â”€â”€ pyvenv.cfg      # Config file
```

### `virtualenv` (Third-party, older but still used)
```bash

```

### `venv` vs `virtualenv`

## 7. Understanding `requirements.txt`

`requirements.txt` = A file that lists all packages your project needs (with versions).

### Creating requirements.txt
```bash
# Auto-generate from current environment
pip freeze > requirements.txt
```

### Example `requirements.txt`
```
requests==2.31.0
flask==3.0.0
sqlalchemy==2.0.23
pytest==7.4.3
```

### Installing from requirements.txt
```bash
# Someone clones your project and runs:
pip install -r requirements.txt
# All packages get installed at the exact same versions!
```

### Why Pin Versions?
```
requests         # âŒ Bad â€” installs whatever latest version exists
requests==2.31.0 # âœ… Good â€” exact version, reproducible builds
requests>=2.28   # âš ï¸ OK for libraries, risky for apps
```

### Workflow for Every New Project
```bash
# 1. Create project folder
mkdir my_project && cd my_project

# 2. Create virtual environment
python3 -m venv venv

# 3. Activate it
source venv/bin/activate

# 4. Install packages
pip install flask requests

# 5. Save dependencies
pip freeze > requirements.txt

# 6. Add venv to .gitignore (NEVER commit venv/)
echo "venv/" >> .gitignore
```

> ğŸ’¡ **Never commit `venv/` to Git.** Only commit `requirements.txt`. Others will recreate the venv using it.

## 8. Understanding Python Versioning (Python 2 vs 3)

### The Big Difference
| Feature | Python 2 (â˜ ï¸ Dead) | Python 3 (âœ… Use This) |
|---------|-------------------|---------------------|
| `print` | `print "hello"` (statement) | `print("hello")` (function) |
| Division | `5/2 = 2` (integer div) | `5/2 = 2.5` (true div) |
| Strings | ASCII by default | Unicode by default |
| `input()` | `raw_input()` | `input()` |
| End of Life | **January 1, 2020** | Actively maintained |

### Python 3 Version History (Key Releases)
| Version | Release | Notable Addition |
|---------|---------|-----------------|
| 3.6 | 2016 | f-strings (`f"Hello {name}"`) |
| 3.7 | 2018 | `dataclasses`, `breakpoint()` |
| 3.8 | 2019 | Walrus operator (`:=`) |
| 3.9 | 2020 | `dict` merge (`\|`), `zoneinfo` |
| 3.10 | 2021 | `match`/`case` (pattern matching) |
| 3.11 | 2022 | 10-60% faster, better errors |
| 3.12 | 2023 | Better f-strings, type hints |
| 3.13 | 2024 | Free-threaded mode (no GIL!) |

### How to Check Your Version
```bash
python3 --version    # Python 3.x.x
python3 -c "import sys; print(sys.version_info)"
```
